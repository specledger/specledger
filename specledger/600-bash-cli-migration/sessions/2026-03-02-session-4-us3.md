**SpecSetupPlanOutput struct**:
```json
{
  "PLAN_FILE": "/path/to/specledger/600-feature/plan.md"
}
```

**Flags implemented**:
- `--json, -j`: JSON output mode

**Command Flow**:
1. Get working directory
2. Detect feature context (repo root, branch, feature directory)
3. Check if plan.md already exists (error if so)
4. Read plan template from embedded FS
5. Create feature directory if needed
6. Write template to plan.md
7. Output results (JSON or text)

**Template Handling**:
- Primary path: `specledger/.specledger/templates/plan-template.md`
- Fallback paths: Multiple alternative locations
- Ultimate fallback: Inline minimal template
- Graceful degradation: Always creates a plan.md

**Error Handling**:
- **Existing plan.md**: Returns error "plan.md already exists at: <path>"
- **Feature context not found**: Returns error with details
- **Template read failure**: Falls back to inline template

**Usage Examples**:

```bash
# Create plan.md in current feature
sl spec setup-plan

# With JSON output
sl spec setup-plan --json

# Example output

Feature: 600-feature-name
```

## Technical Decisions

### 1. Error on Existing Plan
- **Decision**: Return error if plan.md already exists
- **Rationale**: Prevent accidental overwrite of existing work
- **Implementation**: FileExists check before writing

### 2. Template Fallback Strategy
- **Decision**: Multiple fallback paths with inline template
- **Rationale**: Always create a plan.md, even if template missing
- **Implementation**: tryAlternativePlanTemplatePaths() function

### 3. JSON Output Support
- **Decision**: Include --json flag for consistency
- **Rationale**: Match other commands, enable automation
- **Implementation**: SpecSetupPlanOutput struct with JSON encoding
